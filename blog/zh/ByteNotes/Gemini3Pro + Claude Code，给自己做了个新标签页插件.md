## 起因：摸鱼的时候，谁会打开滴答清单啊？

最近在折腾 Chrome 插件开发，顺手给自己做了个小工具。

起因是家人一直用 Momentum，每次新建标签页都能看到今天的任务。我觉得这个设计很聪明——不是让你主动打开 todo app，而是在你准备摸鱼的那一刻拦你一下。

我自己用滴答清单管理任务，但问题是：摸鱼的时候谁会打开滴答啊？打开的是小红书和 B 站。

滴答官方其实有插件，但说实话不够好看。既然都要用，不如自己写一个顺眼的。

于是就有了这个项目：**First Glance**。

## Vibe Coding：Gemini 做设计，Claude Code 写代码

这个项目基本上是 AI 辅助完成的，我来说说具体的分工。

### Gemini 3 Pro：设计参谋

UI 设计这块我不太擅长，所以把 Gemini 当成设计顾问来用。这是最近工作中被同事安利的，但是我和gemini3目前还在合作磨合期(他的上下文会很容易轻易丢失，和claudecode 合作久了会有点对这样的金鱼记忆有点无奈)

这也是我最近颇有感悟的一点：

上班总能被动接触到一些有用的东西，增加一个视角

整体的配色方案是让 Gemini 帮忙出的，当然我也把我的一些个人作品喂给了 gemini，让他大致了解一下我喜欢的风格。

布局上也请教了 Gemini，比如提醒用户的 focus 按钮应该放哪里。任务列表怎么排列等。AI 的建议不一定都对，但能给你一个起点，然后自己再调整。

### Claude Code：开发主力

代码这块主要靠 Claude Code。

核心功能的实现——滴答清单 API 对接、任务列表渲染、番茄钟计时器、主题切换——基本都是和 Claude 协作完成的。遇到 bug 也是丢给它分析，大部分能直接修复。

## 踩坑与反思：AI Coding 不是万能的


### API 文档不全，只能靠猜

滴答清单的官方 API 文档并不完整。比如「收集箱」(inbox) 的接口，文档里根本没提，但实际是存在的。这种情况下 AI 也帮不上忙，只能自己抓包、猜测、调试。

### 边想边改，组件设计容易乱

因为是个人项目，没有提前画原型，想到什么做什么。结果就是 React 组件层级比较混乱，有些地方耦合度高，后期改起来费劲。

这是 Vibe Coding 的一个典型问题：AI 能帮你快速实现功能，但不会主动帮你规划架构。如果你自己没想清楚，代码很容易写成一坨。

### 性能问题需要人工把关

AI 生成的代码有时候不够优化，比如不必要的重渲染、state 管理不合理。这些问题 AI 不会主动提醒你，需要自己有意识地去 review。

**我的建议**：
- 先花时间想清楚要做什么，画个简单的原型或写个需求文档
- 定期停下来看看代码结构，别只顾着加功能
- 性能敏感的地方自己过一遍

## 功能展示

最后介绍一下做出来的东西。

**Focus 模式**：极简界面，一个时钟 + 今日 Top 3 任务。打开新标签页就能看到最重要的事，帮你保持专注。

![[Pasted image 20260122084909.png]]

**List 模式**：完整的任务管理视图。Today、Tomorrow、本周、过期、收集箱，分类清晰。可以直接在这里完成、编辑、删除、新建任务。
![[Pasted image 20260122084930.png]]


**多种主题**：7 款主题可选，从粉色到薄荷色，深色/浅色模式自适应。

![[Pasted image 20260122084952.png]]

**番茄钟**：内置 25 分钟工作 + 5 分钟休息的计时器，跨标签页同步。

**游客模式**：不登录滴答清单也能用，最多管理 3 个本地任务。
