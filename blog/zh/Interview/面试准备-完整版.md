# 面试准备

## 自我介绍（1-2 分钟）

我是一名有 4 年经验的前端开发，之前在字节跳动和商汤都做过项目。

我不是计算机专业出身，但入职半年就涨薪 50%，后来也独立负责过几个完整的项目。所以学习能力这块，我觉得用结果说话比较有说服力。

过去一年多我有一段职业空白期。这段时间我没闲着——刷了不少 LeetCode，做了一些个人项目，也花了很多时间思考自己到底想要什么样的工作。

我对这个岗位感兴趣，是因为它不只看技术，还看产品感和审美。这正好是我这段时间一直在积累的方向。

## 必答题

### 1. 你有一段空窗期，能说说吗？

这 18 个月是我主动选择的调整期。

离开上一份工作的时候，我意识到自己在处理压力这件事上有问题。倒不是技术能力不够，而是我会把项目的困难等同于"我不行"——这其实是一种认知上的扭曲。

所以我停下来，认真学了一些心理学的东西，比如认知行为疗法。我现在能更清楚地区分：项目遇到困难是客观事实，但它不能定义我这个人。

这段时间我也没停下技术学习。LeetCode 刷了几十道，做了个人博客系统，还写了一些自动化工具。

现在回头看，这段经历让我更清楚自己想要什么，也更有能力在压力下保持冷静。

### 2. 你不是计算机专业，怎么转的？

对，我不是科班出身。但我觉得这反而让我有一个习惯——我学东西的时候会更关注"为什么这样设计"，而不是"就是这样"。

比如学 React，我会先去理解它的设计哲学：声明式、组件化、单向数据流。理解了这些，具体的 API 用起来就顺了。

我有自己的学习方法：建立心智模型比记忆语法更重要。宁愿花一周时间把一个概念搞透，也不要快速刷过十个概念，回头全忘了。

至于能力，我觉得用结果说话比较直接——入职半年涨薪 50%，后来也独立带过项目。

### 3. 上一份工作为什么离开？

直接原因是项目遇到了困难，但根本原因是我当时不会处理这种压力。

我把项目的问题内化成了对自己的否定。现在我知道这叫"认知扭曲"。项目失败是客观事实，但它不等于"我不行"。

这也是为什么 gap 这段时间对我很重要——如果不停下来想清楚这个问题，以后遇到类似情况还是会重蹈覆辙。

### 4. 你觉得你适合这个岗位吗？

我觉得挺适合的，虽然我之前没有正式的产品或 UI 工作经验。但这段 gap 期间积累的东西，反而可能是我的优势。

**关于审美：**

我读了原研哉的《设计中的设计》和《Ex-Formation》。给我印象最深的一句话是："知道是想象的起点，而非终点。"如果对话中充斥着"我知道"，正说明我们在不知不觉间脱离了把知识升华为想象的轨道。

我也去看展览。比如之前在杭州看徐渭的画，站在那些作品前面，我会想：他几百年前的情绪，为什么现在还能打动我？这种思考对我来说是自然的。

**关于用户思维：**

我一直在用各种 AI 编程工具，Claude Code、GPT 都用。用多了之后我发现一个问题：当工具让事情变得太容易，人就会停止思考。我把这个叫"便捷陷阱"。

我写过一篇文章分析这个问题。本质上，这就是在用同理心去理解用户（包括我自己）的行为模式。好的产品设计不是给用户想要的，而是帮用户解决他们可能还没意识到的问题。

**关于产品能力：**

我做过一些从零到一的小项目。比如我的个人博客，我没有去搭一个完整的后台编辑系统，而是直接用 Obsidian 写、Git 同步、GitHub API 拉取、前端渲染。

为什么这样设计？因为 Obsidian 的编辑体验已经很好了，我没必要重复造轮子。这就是产品思维——不是炫技，而是用最简单的方式解决问题。

## 技术项目

### 项目一：大文件分片上传系统

**背景：**

之前做运维系统，需要上传几百 MB 到几 GB 的 OTA 固件包。传统单次上传有几个问题：浏览器内存溢出、网络中断要从头来、用户不知道进度到哪了。

**我的方案：**

把大文件切成 5MB 的分片，用 Web Worker 在后台线程处理，避免页面卡死。

每个分片单独计算 MD5，用 SparkMD5 增量计算整体哈希。服务端返回已上传的分片列表，客户端跳过这些，实现断点续传。

并发控制在 3 个，因为浏览器对同域名有 6 个连接数限制，留一些给其他请求。

**亮点：**

- Web Worker 处理 CPU 密集型操作，不阻塞 UI
- Set 存储已上传分片，O(1) 查找
- requestAnimationFrame 轮询队列，比 setInterval 更节能

### 项目二：WebSocket 跨框架封装库

**背景：**

原生 WebSocket 缺乏自动重连、消息缓冲、多连接管理、心跳检测。我们需要同时支持 React 和 Vue。

**我的方案：**

核心逻辑和框架无关，只有顶层 Hook 使用框架特定 API。

用有限状态机管理连接生命周期：None → Inited → Opened → Sent/Received → Closing → Closed。

实现了消息缓冲：连接未就绪时自动缓存，环形缓冲区最多 20 条，连接建立后自动 flush 最新消息。

**为什么只发送最新消息而不是全部？**

因为对于实时数据场景，旧消息已经过时了。这是针对业务场景的 trade-off。

### 项目三：邮件同步系统

**背景：**

个人项目，把 Outlook 邮件同步到本地 MongoDB，支持按联系人过滤、按标记筛选。

**技术栈：**

Next.js + Microsoft Graph API + MongoDB + Cloudinary

**亮点：**

- 设备码认证流程（Device Code Flow）
- 邮件分离正文和引用部分
- 内联图片自动上传到 Cloudinary
- 前端实时进度展示

这个项目体现的是全栈能力，从 OAuth 认证到数据处理到前端展示，都是我一个人做的。

## 软实力

### AI 工具使用

我用 AI 编程工具挺多的，Claude Code 和 GPT 都用。但我有一个原则：**AI 应该自动化重复工作，而不是替代深度思考。**

我发现一个问题：当工具让事情变得毫不费力，人就会停止专注。这就是"便捷陷阱"。

所以我用 AI 的方式是：让它帮我处理重复性的工作，比如写脚本、生成样板代码。但架构设计、技术选型这些需要判断力的事情，我会自己来。

**一个例子：**

我写了一个 Git Worktree 自动化脚本，从 v1 迭代到 v4。每次迭代都是因为实际使用中发现了新的痛点：v2 加了 .env 自动同步，v3 支持分支名带斜杠，v4 加了开发分支指示器。

AI 帮我写代码，但发现问题、定义需求是我自己做的。

### 学习方法

我有一个习惯：**输出倒推输入**。

今年我参加了一个读书会。我发现很多人只看目录，然后让 AI 生成总结。但我觉得这本末倒置了。

最重要的不是信息的输入，而是尝试把书里的知识和自己脑子里已有的知识关联起来。这个过程很慢，但对自己是真正有益的。

所以我会强迫自己写文章。写的过程中，我需要把零散的想法组织成逻辑链条，这本身就是在加深理解。

### 设计观点

我读过原研哉的书，有几句话印象很深：

- "知道是想象的起点，而非终点"
- "创意不是让人惊讶于它崭新的形式，而是让人惊讶于它居然来自看似平凡的日常"
- "新技术的出现不是让新的取代旧的，而是旧的要容纳新的"

这些观点影响了我对产品和设计的理解。好的设计不是炫技，而是在平凡中发现价值。
