**问题的根源**

我觉得很多用 Claude Code的人都遇到过这个问题：当 Claude在处理问题时 ，我们可能会去做其他事情。当他在执行可能存在风险的操作前，会暂停并等待我们的许可——我发现，这可能正是问题出现的时刻。

我有时候会上网冲浪，在互联网上寻求多巴胺刺激时，可能会被其内容深深吸引。这可能导致我快速浏览 Claude 的计划或警告消息，我最终在没有仔细思考的情况下按下回车键。

后来，当我遇到 Claude 多次尝试但未能修复的 bug 时，真实的问题才显现出来：浪费的时间和 token——更重要的是，开发过程中的势头被打破了。

**我的解决方案：双维度日志记录**

因此，我这次 Claude Code 编写了一个脚本，在每个任务完成时将其总结为 markdown 文件。这创建了所做工作的自动记录。主要实现的原理在于利用 claudecode 的 hooks 生命周期，在 stop 时读取本地的 jsonl 会话文件，提取完成的任务总结

这个日志记录器在两个维度上运作。

- 第一个维度按时间顺序总结——如果你想查看今天发生了什么，只需查看今天的日志文件。
- 第二个维度按项目——可以追踪项目日志来理解做了哪些更改以及顺序是什么。

**额外的收获**

当我请求 Claude 修改我的英文草稿时，它不仅完成了我的要求，还提供了深思熟虑的改进建议。这提醒我，工具只是辅助，其实真正重要的还是自己掌控专注力。

**管理注意力的策略**

**在任务执行间**，尽可能让终端 pin 在顶端，抵抗切换注意力的冲动。保持专注于你的终端或编辑器。

**将工作分批进行**，进行专注的编码会话，避免其他诱人的标签或应用程序。每个人的大脑注意力资源有限，不应该分散在多个上下文中。

**主动阅读计划**再进行批准——考虑添加一个有意的 5 秒暂停，确保你理解 Claude 提议做什么。
