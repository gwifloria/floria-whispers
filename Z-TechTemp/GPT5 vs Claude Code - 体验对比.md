
## GPT5

### 使用要求
- **需要明确指令**：不能给比较笼统的指令，否则随意发挥空间太大
- **后续修改成本高**：根据其自由发挥再修修改改容易堆积屎山

### 性能体验
- **响应速度**：等待过程是中断的，碎片化时间利用是大问题，速度比较慢

### 使用风险
- **思考退化**：会让人依赖且思考退化，可能甚至会懒得查询文档，而选择直接问 GPT
- **代码质量控制**：完成一个功能点后需要及时确认
  - 肉眼看代码会不会出现一些古早写法
  - 单元测试、e2e测试及时生成
  - 等生成了一大堆后再想去修改会是一个大问题

### 适用场景
- 一些枯燥的工作可以搞定
- 但也要时刻去反思有没有更好的写法

### 技术问题
- **感觉不如 GPT-4 聪明**，想退钱
- **React Hooks 错误**：每次都把 hooks 放在条件判断后写给我，真是太蠢了，告诉他几遍了，下次写代码依旧如此

### 信息准确性问题
- **会幻想一些东西出来**
- **Microsoft 365 E5 案例**：
  - 在做 Outlook 邮件抓取时，建议我去注册 Microsoft 365 E5 开发者
  - 我特地问了注册地区是否有要求（比如避开中国大陆）
  - 在明确告知我可行之后我才进行了相关注册
  - 但发现依旧看不到 E5 sandbox 功能后，我去小红书搜索了一下
  - 发现这个功能在大陆地区就是被 ban 的
  - 回头去和 GPT5 对话，他又告诉我不行
- **很爱幻想的小伙子**

## Claude Code

### 技术偏好问题
- **Next.js Image 组件**：比如项目中有 Next.js，引入 Image 会默认写 `/img`，改成 Image 后会优先写 deprecated 的属性

### 交互特点
- **对话风格**：和 Claude 对话有点像在和老妈子对话，很啰嗦
- **解决问题的态度**：
  - 当我碰到一个问题时候：比如电脑的网络配置问题，我的电脑默认 DNS 解析可能被污染了，导致我 Claude 和 Obsidian 联动有点问题
  - 其实 DNS 污染可能是一个也需要解决的问题
  - 越沟通越喜欢花大量篇幅告诉我"我们不要解决 DNS 解析了，找一个办法 workaround"

### 亮点功能
- **`/planmode` 是一个好东西**：
  - 其实也是一个好习惯
  - 比起拿到需求先不动脑子开发
  - 不如先花点时间想想整个流程是什么，优先级如何，会遇到什么问题

## 总结

两个工具各有特点：
- **GPT5** 适合明确任务，但需要严格控制和及时验证
- **Claude Code** 更适合规划和探索，但技术细节需要注意
- 都需要保持独立思考，避免过度依赖
