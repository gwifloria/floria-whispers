# 为什么开始做个人网站

过去一直没有做个人网站。回顾原因，大概有几点：

1. **记录习惯**
   - 我平时会写一些日记，但只是作为个人记录，并没有考虑过公开。
   - 如果要公开，就会涉及到 **编辑功能**，而我已经习惯用 Obsidian 来记录。
   - 如果要兼顾两端同步，可能需要额外开发一套编辑/存储方案，这让我觉得很麻烦。

2. **心理障碍**
   - 我可能 **过度放大了过程的麻烦程度**，把“可能会碰到的问题”都想得很严重，于是干脆不开始。
   - 习惯于“完美主义”思维，想“一口气吃成胖子”，觉得要一次性解决所有问题才行。
   - 于是拖延成了常态：脑子里越想，问题就越复杂，反而更难下手。

3. **写作的断层**
   - 高考前，我们都在写作文，写作是 **观察 → 感受 → 思考 → 输出** 的过程。
   - 停止写作之后，我也少了很多深度思考的机会。
   - 每一次写作，能把零散的灵感与已有知识连接起来，这才是长期的“知识财富”。
   - 所以没有写作，也让我觉得自己停滞了。

---

## 解决思路

后来我尝试把问题拆解，发现其实没有那么难：

1. **前端与内容分离**
   - 我不希望前端展示项目里夹杂大量 `.md` 文件。
   - 因此决定利用 **Obsidian 的自动提交插件**，把笔记同步到一个独立的 Git 仓库。
   - 前端项目只需要去请求这个仓库的内容，并展示指定目录下的 Markdown。

2. **内容筛选**
   - 我的 Obsidian 仓库里有很多半成品、随笔不适合公开。
   - 所以我把 **公开内容放在一个指定的文件夹** 里，前端只展示这个目录的内容。
   - 这样既保持了写作习惯，又能保证前端内容的干净、统一。

3. **仓库管理**
   - 这个公开内容的仓库会设置成 **私有**。
   - 前端访问时，借助 GitHub API 和 `GITHUB_TOKEN` 来拉取内容。
   - 这样我就不需要实现编辑功能，保持“只读”展示即可。

4. **自动化与优化**
   - 平时可以手动 `git pull` 保持前端同步。
   - 后续可以写一个脚本，或者用 CI/CD 定时拉取更新。

---

## GitHub 接口与 Token

1. **API 限制**
   - GitHub API 默认的请求次数限制大概是 **60 次/小时（未授权）**。
   - 如果加上 `Authorization: Bearer GITHUB_TOKEN`，配合私有仓库访问，就能提高限制并访问受保护的内容。

2. **实践**
   - 在开发调试时，必须申请 `GITHUB_TOKEN`，并配置到环境变量中。
   - 前端通过调用自己的 `/api/gh-content` 代理，再去请求 GitHub API，避免 token 直接暴露在客户端。

3. **保持最新**
   - 我的前端不会直接读本地 `.md` 文件，而是调用 **Next.js API**。
   - Next API 每次请求时直接去 GitHub 仓库取内容，这样保证每次展示的都是最新的。
   - 未来可以在 Next API 层加上缓存（例如 60s revalidate），兼顾性能与实时性。

---

## 总结

这个过程的关键其实是 **降低启动门槛**。  
- 不必重造编辑器，不必一开始就做全功能。  
- 保持 **写在 Obsidian → 自动提交 → GitHub 仓库 → 前端展示** 的闭环。  
- 从最小化 MVP（只读博客）开始，后续再逐渐完善。  

写作的意义在于：**当一个人有思考，就会有输出。**
