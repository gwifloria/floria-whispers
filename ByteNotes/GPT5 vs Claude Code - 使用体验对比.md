
> 基于实际开发工作中的使用体验对比

## 总体评价概览

| 特性   | GPT5            | Claude Code |
| ---- | --------------- | ----------- |
| 指令理解 | 需要明确具体的指令       | 更好的上下文理解    |
| 响应速度 | 较慢，等待时间碎片化      | 相对更快        |
| 代码质量 | 容易过度发挥，需要频繁修正   | 有特定技术栈的固有偏好 |
| 工作流程 | 适合修一些页面小bug 小样式 | 更适合整体功能开发   |

## GPT5 使用体验

### 优势
- **在使用 CC后，想不出来**
- 现在会喜欢用免费版本
	- 指定文件浅浅修改一下样式，省点钱
	- 问一些碎片化的问题或帮忙整理一下 md 格式

### 痛点与问题

#### 1. 响应体验问题
**问题**：建立 project 代码后，回复速度会越来越慢，并且 cpu 占比极高
**影响**：
- 响应速度相对较慢
- 等待时间碎片化，cpu 占比高的情况下电脑也没法用于做其他是，最近会在面前摆一本书，比刷手机号

#### 2. 代码质量
**React Hooks 问题**：
- 经常将 hooks 放在条件判断后面
- 多次提醒后仍会重复相同错误

#### 3. 信息准确性问题
**Microsoft 365 E5 开发者注册案例**：
-  我在实现 outlook 邮件抓取功能时，gpt5建议注册 E5 开发者账号做邮件抓取
- 我在明确询问地区（大陆）限制时，他告诉我大陆是 accessible 的
- 后续我在其他论坛搜索发现这个再大陆地区是禁用 sandbox 的
- 后续询问时又改口说不可行

**总结**：容易提供不准确或过时的信息

### 适用场景
- 明确的、具体的开发任务
- 不需要探索性的重复性工作
- 需要严格按照规范的代码生成

## Claude Code 使用体验

### 优势
- `/planmode` 功能有助于项目规划，仔细规划完后，可以直接放手去思考下一步完整的功能点。
- 可中断，发现当前任务执行偏离自己期望后，可以 esc 停止

### 痛点与问题

#### 1. 技术栈偏好问题
**Next.js Image 组件问题**：
- 引入 Image 组件时默认写成 `/img` 路径
- 修正为正确的 `Image` 后优先使用已废弃的属性
- 对特定技术栈存在固化的错误偏好

#### 2. 交互方式问题
**过度谨慎**：
- 对话风格过于谨慎和冗长
- 倾向于规避问题而非解决问题

**实际案例 - DNS 配置问题**：
- 遇到网络配置问题时
- 明显的 DNS 污染需要解决
- 但倾向于建议 workaround 而非根本解决
- 花大量篇幅劝说避免解决根本问题

#### 3. 开发流程优势
**Plan Mode 的价值**：
- 强制在开发前进行思考
- 帮助梳理整体流程
- 明确优先级和潜在问题
- 比直接开始编码更有效

### 适用场景
- 需要项目规划的复杂任务
- 探索性开发工作
- 需要反复讨论和调整的需求

---

### 通用注意事项
1. **及时验证**：无论使用哪个工具，都要及时验证生成的代码和信息
2. **保持主动**：避免过度依赖，保持独立思考和查阅文档的习惯
3. **分段确认**：完成一个功能点后及时确认，避免问题累积


---

## 个人总结

两个工具都有各自的优势和局限性，关键在于：
1. 根据任务特点选择合适的工具
2. 保持对生成内容的critical thinking
3. 不要完全依赖AI，保持技术判断能力
4. 将AI作为辅助工具而非替代方案

5. 尽可能给出明确指令，gpt 毕竟不是自己的大脑，可能大脑里有些细节是已经默认存在于脑海之中，甚至想不起来需要去告诉对方。大家对“美”“可爱”的定义都是不同的，需要给到客观称述，而不是一个抽象词汇

ai 工具相当于是一个当今世界大家共同的知识图谱，但是具体怎么关联，还是要靠自己去下命令，才能形成属于自己的知识图谱

*最重要的是保持学习新技术和解决问题的能力，而不是让AI成为思考的替代品。*
